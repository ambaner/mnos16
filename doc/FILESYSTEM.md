# MNFS — Mini-OS Flat Filesystem Specification

**Version:** 1.1
**Status:** Design — v0.9.11

---

## 1. Overview

MNFS (Mini-OS Flat Filesystem) is a minimalist, flat (no subdirectories) filesystem
designed for mini-os.  Its primary goals are:

- **Eliminate hardcoded disk offsets** — all binaries (LOADER, KERNEL, FS, SHELL)
  are located by name through a directory table, not fixed sector numbers.
- **Modular filesystem driver** — the filesystem is implemented as a loadable
  kernel module (`FS.SYS`) that exposes its own software interrupt (`INT 0x81`),
  fully decoupled from the kernel.
- **Educational clarity** — the design mirrors real-world concepts (directory
  tables, filesystem drivers, layered APIs) while remaining simple enough to
  understand in a single reading.

MNFS supports up to 15 files in a single flat directory.  Files are stored as
contiguous sectors — there is no fragmentation, no allocation table, and no
subdirectory support.

---

## 2. Design Goals

| Goal                 | How MNFS achieves it                                     |
|----------------------|----------------------------------------------------------|
| No hardcoded offsets | Directory entries store each file's start sector         |
| Single source of truth | The directory table on disk is the only authority       |
| Modular FS driver    | `FS.SYS` is a separate loadable module with its own ISR  |
| Layered I/O          | FS.SYS uses the kernel's `INT 0x80` for disk I/O         |
| Simple build pipeline| `create-disk.ps1` generates the directory automatically  |
| Real-world patterns  | 8.3 filenames, file attributes, contiguous allocation    |

---

## 3. Disk Layout

The partition is divided into three regions: the VBR, the MNFS directory, and
the file data area.  The VBR and directory are at fixed, well-known offsets.
All file positions are dynamic (recorded in the directory).

```
Partition-relative sector map
──────────────────────────────────────────────────────────────────────
Sector 0-1      VBR (2 sectors, 1 KB)
                  - MNOS magic, sector count, partition LBA
                  - Bootstrap code: reads directory, finds LOADER

Sector 2        MNFS Directory Table (1 sector, 512 bytes)
                  - 32-byte header + up to 15 × 32-byte file entries
                  - Generated by create-disk.ps1 at build time

Sector 3+       File Data Area (files packed contiguously)
                  - LOADER.SYS   (2 sectors,  1 KB)
                  - FS.SYS       (2 sectors,  1 KB)
                  - KERNEL.SYS   (4 sectors,  2 KB)
                  - SHELL.SYS   (10 sectors,  5 KB)
                  - (future files appended here)
──────────────────────────────────────────────────────────────────────
```

The exact start sector of each file is determined by `create-disk.ps1` at
build time and written into the corresponding directory entry.  Files are
packed contiguously with no gaps.  The only hardcoded constant is the
directory table location at partition sector 2 (`MNFS_DIR_SECTOR equ 2`).

### 3.1 Example Layout (current build)

| Partition Sector | Content         | Sectors | Size   |
|-----------------|-----------------|---------|--------|
| 0-1             | VBR             | 2       | 1 KB   |
| 2               | MNFS Directory  | 1       | 512 B  |
| 3-4             | LOADER.SYS      | 2       | 1 KB   |
| 5-6             | FS.SYS          | 2       | 1 KB   |
| 7-10            | KERNEL.SYS      | 4       | 2 KB   |
| 11-20           | SHELL.SYS       | 10      | 5 KB   |

## 4. Directory Table Format

The MNFS directory occupies exactly 1 sector (512 bytes).  It begins with a
32-byte header, followed by up to 15 directory entries of 32 bytes each.

```
Offset  Size  Field           Description
──────  ────  ──────────────  ────────────────────────────────────
0x00    4     magic           'MNFS' (4 bytes, ASCII)
0x04    1     version         Filesystem version (0x01)
0x05    1     file_count      Number of valid entries (0-15)
0x06    2     total_sectors   Total sectors used (directory + all files)
0x08    2     capacity        Partition data capacity in sectors (partition size minus VBR)
0x0A    22    reserved        Reserved for future use (zero-filled)
──────  ────  ──────────────  ────────────────────────────────────
0x20    32    entry[0]        First directory entry
0x40    32    entry[1]        Second directory entry
...
0x1E0   32    entry[14]       Fifteenth (last) directory entry
```

**Magic validation:** Any code reading the directory MUST verify the first
4 bytes are `MNFS` (0x4D 0x4E 0x46 0x53) before trusting the contents.

---

## 5. Directory Entry Format

Each directory entry is 32 bytes:

```
Offset  Size  Field           Description
──────  ────  ──────────────  ──────────────────────────────────────────────
0x00    8     filename        Filename, uppercase, space-padded (e.g., "LOADER  ")
0x08    3     extension       Extension, uppercase, space-padded (e.g., "SYS")
0x0B    1     attributes      Bit flags (see below)
0x0C    4     start_sector    Partition-relative start sector (uint32 LE)
0x10    2     size_sectors    File size in sectors (uint16 LE)
0x12    4     size_bytes      Actual file size in bytes (uint32 LE)
0x16    10    reserved        Reserved (zero-filled)
```

### 5.1 Attributes Byte

| Bit | Name       | Meaning                                       |
|-----|------------|-----------------------------------------------|
| 0   | SYSTEM     | System file — required for boot (LOADER, KERNEL, FS) |
| 1   | EXECUTABLE | User-mode executable (MNEX format)             |
| 2-7 | Reserved   | Must be zero                                   |

Example attribute values:
- `0x01` — System file (LOADER.SYS, KERNEL.SYS, FS.SYS, MM.SYS)
- `0x03` — System executable (SHELL.SYS — system-loaded at boot, but executable)
- `0x02` — User executable (future `.MNX` user programs)
- `0x00` — Data file (future use)

> **Why is SHELL.SYS marked SYSTEM+EXECUTABLE (0x03)?**  The shell is loaded by
> the kernel at boot and lives in system memory, making it a system component
> (`ATTR_SYSTEM`).  But it is also a program — the command interpreter — so it
> carries `ATTR_EXEC` as well.  The combined attribute displays as "SYS+X" in
> the `dir` command.  When user-mode loading is implemented (v1.0.0), the
> `ATTR_SYSTEM` bit will prevent the shell from being accidentally `run` from
> the command line, providing protection alongside the `.SYS` extension check.

### 5.2 Filename Matching

Filenames are compared as 11-byte blocks (8 name + 3 extension).  Both the
directory entry and the search key must be uppercase, space-padded, with no
dot separator.  Examples:

| Human-readable | 11-byte directory form     |
|----------------|----------------------------|
| `LOADER.SYS`   | `LOADER  SYS`              |
| `KERNEL.SYS`   | `KERNEL  SYS`              |
| `FS.SYS`       | `FS      SYS`              |
| `SHELL.SYS`    | `SHELL   SYS`              |
| `README.TXT`   | `README  TXT`              |

This is identical to the FAT12/16 8.3 directory entry format.

## 6. Bootstrap vs. Runtime Access

A key architectural question: **how do boot-time components (VBR, LOADER,
KERNEL) read the directory before FS.SYS is loaded?**

The answer is a two-layer design:

### 6.1 Bootstrap Layer (boot-time, inline code)

VBR, LOADER, and KERNEL each `%include "find_file.inc"` — a small (~80 byte)
shared subroutine that reads the MNFS directory sector from disk using raw
INT 0x13h and searches it for a given filename.

This is the "bootstrap filesystem reader" — just enough to find files by name.
It mirrors how real bootloaders contain minimal filesystem readers (GRUB reads
ext4/FAT, Windows bootmgr reads NTFS) even though the full driver is loaded
later.

```
Boot-time flow:
  VBR:    find_file("LOADER  SYS")  → loads LOADER
  LOADER: find_file("KERNEL  SYS")  → loads KERNEL
  KERNEL: find_file("FS      SYS")  → loads FS.SYS
          find_file("SHELL   SYS")  → loads SHELL
```

### 6.2 Runtime Layer (after boot, FS.SYS module)

Once the kernel loads `FS.SYS` and calls its init routine, the filesystem
module installs an `INT 0x81` handler.  From that point on, user-mode
programs (SHELL) access the filesystem through `INT 0x81` syscalls — they
never touch INT 0x13h or the raw directory format directly.

```
Runtime flow:
  Shell "dir" command:
    INT 0x81 (AH=0x01)   →   FS.SYS returns directory data
    Shell formats and displays
```

### 6.3 Why Two Layers?

| Concern         | Bootstrap (find_file.inc) | Runtime (FS.SYS INT 0x81) |
|-----------------|--------------------------|---------------------------|
| When available  | Always (compiled in)     | After kernel loads it     |
| Code size       | ~80 bytes per binary     | Full module, ~1 KB        |
| Disk I/O        | Direct INT 0x13h         | Via kernel INT 0x80       |
| Used by         | VBR, LOADER, KERNEL      | SHELL (user-mode)         |
| Purpose         | Find and load one file   | Full FS API for apps      |

This is the same pattern as real operating systems: the bootloader has a tiny
filesystem reader, and the full filesystem driver is loaded as part of the OS.

## 7. FS.SYS — Filesystem Module

### 7.1 Identity

| Property        | Value                                            |
|-----------------|--------------------------------------------------|
| Magic           | `MNFS` (4 bytes)                                 |
| Header          | 6 bytes: magic (4) + sector_count (2)            |
| Load address    | 0x0800 (reuses LOADER.SYS's memory after boot)   |
| Loaded by       | KERNEL, via `find_file("FS      SYS")`           |
| Interrupt       | INT 0x81                                         |
| Disk I/O        | Via kernel's INT 0x80 `SYS_READ_SECTOR`          |

### 7.2 Memory Placement

After the boot chain completes, LOADER.SYS's memory at 0x0800 is no longer
needed.  The kernel loads FS.SYS into this same address, reusing the space:

```
Runtime memory map:
  0x0600  Boot Info Block (BIB)       6 bytes
  0x0800  FS.SYS (was LOADER.SYS)    up to 8 KB
  0x3000  SHELL.SYS                  up to 8 KB
  0x5000  KERNEL.SYS                 up to 8 KB
  0x7C00  (VBR, freed at boot)       —
  ↓       Stack grows downward       from 0x7C00
```

### 7.3 Initialization

The kernel calls FS.SYS's init entry point after loading:

```
KERNEL:
  1. find_file("FS      SYS")         → get disk location
  2. load_mnex(...)                    → load FS.SYS at 0x0800
  3. call FS_BASE + 6                  → call FS init routine
  4. find_file("SHELL   SYS")         → get disk location
  5. load_mnex(...)                    → load SHELL at 0x3000
  6. jmp SHELL                         → start shell
```

FS.SYS's init routine:
1. Installs the INT 0x81 vector in the IVT (at 0x0000:0x0204)
2. Reads the MNFS directory sector via `INT 0x80 SYS_READ_SECTOR` into an
   internal 512-byte cache
3. Validates the `MNFS` magic in the cached directory
4. Returns to the kernel (CF clear = success, CF set = error)

After init, the directory is cached in FS.SYS's memory.  Subsequent
`INT 0x81` calls are served from cache without additional disk I/O
(except `FS_READ_FILE` which reads file data from disk).

### 7.4 Binary Layout

```nasm
; FS.SYS memory layout (loaded at 0x0800)
;
; Offset 0x00:  'MNFS' magic + sector count (6 bytes)
; Offset 0x06:  fs_init entry point
; Offset ????:  INT 0x81 handler + dispatch
; Offset ????:  FS function implementations
; Offset ????:  512-byte directory cache
; Offset ????:  Data/strings
```

## 8. INT 0x81 — Filesystem Syscall Interface

FS.SYS installs an interrupt handler at vector 0x81 (IVT address 0x0204).
User-mode programs invoke filesystem operations via `INT 0x81` with the
function number in `AH`.

This mirrors how the kernel uses `INT 0x80` for system calls.  The separation
into two interrupt vectors provides clean architectural layering:

- **INT 0x80** — Kernel services (video, keyboard, memory, disk I/O)
- **INT 0x81** — Filesystem services (directory listing, file lookup, file read/write/delete/rename)

### 8.1 Function Table

| AH  | Name            | Description                          |
|-----|-----------------|--------------------------------------|
| 0x01| FS_LIST_FILES   | Return the cached directory data     |
| 0x02| FS_FIND_FILE    | Find a file by 8.3 name              |
| 0x03| FS_READ_FILE    | Read a file's contents into memory   |
| 0x04| FS_GET_INFO     | Get filesystem metadata              |
| 0x06| FS_WRITE_FILE   | Write/create a file on disk          |
| 0x07| FS_DELETE_FILE   | Delete a file (tombstone)            |
| 0x08| FS_RENAME_FILE  | Rename a file                        |

### 8.2 FS_LIST_FILES (AH=0x01)

Returns the raw cached directory sector to the caller's buffer.

```
Input:
  AH    = 0x01
  ES:BX = pointer to 512-byte buffer

Output:
  CL    = file count (from directory header)
  CF    = clear (always succeeds — data is cached)
  Buffer filled with 512 bytes of raw MNFS directory data
```

The caller is responsible for parsing the directory entries from the buffer.
This keeps FS.SYS simple and gives user-mode programs full flexibility in
how they display directory contents.

### 8.3 FS_FIND_FILE (AH=0x02)

Searches the cached directory for a file by its 11-byte 8.3 name.

```
Input:
  AH    = 0x02
  DS:SI = pointer to 11-byte filename (8 name + 3 ext, space-padded, uppercase)

Output:
  CF clear = found:
    EAX = start sector (partition-relative)
    CX  = size in sectors
    EDX = size in bytes
    BL  = attribute byte (v0.9.6+)
  CF set   = not found
```

### 8.4 FS_READ_FILE (AH=0x03)

Reads a file's contents from disk into the caller's buffer.  Internally,
FS.SYS calls `FS_FIND_FILE` to locate the file, then uses the kernel's
`INT 0x80 SYS_READ_SECTOR` to read the data.

```
Input:
  AH    = 0x03
  DS:SI = pointer to 11-byte filename
  ES:BX = buffer to read into
  CX    = maximum sectors to read

Output:
  CF clear = success:
    CX  = sectors actually read
  CF set   = error (file not found or disk I/O error)
```

### 8.5 FS_GET_INFO (AH=0x04)

Returns filesystem metadata including partition capacity for space calculations.

```
Input:
  AH = 0x04

Output:
  AL = MNFS version (0x01)
  CL = file count
  CH = max entries (15)
  DX = total sectors used by all files + directory
  BX = partition data capacity in sectors (from MNFS_HDR_CAPACITY)
```

The shell uses DX and BX to calculate and display used/free space statistics.

### 8.6 FS_WRITE_FILE (AH=0x06)

Writes (creates or overwrites) a file on disk.  If the file already exists, it
is overwritten in place (must fit in existing allocation) or an error is
returned.  If it does not exist, a new directory entry is allocated and sectors
are appended after the current high-water mark.

```
Input:
  AH    = 0x06
  DS:SI = pointer to 11-byte filename (8 name + 3 ext, space-padded, uppercase)
  ES:BX = pointer to file data
  ECX   = file size in bytes

Output:
  CF clear = success
  CF set   = error:
    AL = error code (see §8.9)
```

The directory header's `total_sectors` field is updated to reflect the new
high-water mark after a successful write.

### 8.7 FS_DELETE_FILE (AH=0x07)

Deletes a file by marking its directory entry as a tombstone.  The first byte
of the filename is overwritten with `0xE5` (see §8.10 Tombstone Semantics).
System files (ATTR_SYSTEM) are protected and cannot be deleted.

```
Input:
  AH    = 0x07
  DS:SI = pointer to 11-byte filename

Output:
  CF clear = success
  CF set   = error:
    AL = error code (see §8.9)
```

### 8.8 FS_RENAME_FILE (AH=0x08)

Renames a file by updating the 11-byte name in its directory entry.  The new
name must not already exist in the directory.

```
Input:
  AH    = 0x08
  DS:SI = pointer to old 11-byte filename
  ES:DI = pointer to new 11-byte filename

Output:
  CF clear = success
  CF set   = error:
    AL = error code (see §8.9)
```

### 8.9 Error Codes

When a write operation fails (CF=1), AL contains one of these error codes:

| Code | Constant          | Meaning                                    |
|------|-------------------|--------------------------------------------|
| 1    | FS_ERR_NOT_FOUND  | File not found (delete/rename target)      |
| 2    | FS_ERR_EXISTS     | File already exists (rename destination)   |
| 3    | FS_ERR_DIR_FULL   | Directory full (15 entries, no tombstones) |
| 4    | FS_ERR_DISK_FULL  | Not enough free sectors for file data      |
| 5    | FS_ERR_IO         | Disk I/O error during read or write        |
| 6    | FS_ERR_PROTECTED  | File is system-protected (ATTR_SYSTEM)     |

### 8.10 Tombstone Semantics

Deleted files use a **tombstone** marker rather than compacting the directory:

- When a file is deleted, `name[0]` is set to `0xE5` (the FAT-compatible
  tombstone byte).  All other entry fields are preserved.
- Directory scans (FS_LIST_FILES, FS_FIND_FILE) skip entries where
  `name[0] == 0xE5`.
- FS_WRITE_FILE reuses tombstoned entries when allocating new directory slots.
- The `file_count` in the directory header is decremented on delete.
- `total_sectors` is recalculated as the high-water mark (highest
  `start_sector + size_sectors` of any live entry) — it does not shrink to
  fill gaps left by deleted files because MNFS uses contiguous allocation.

### 8.11 Disk Space Management

MNFS uses a simple contiguous allocation model with high-water-mark tracking:

- New files are written at the sector immediately after the last used sector
  (tracked by `total_sectors` in the directory header).
- Deleted file sectors become unreclaimable gaps — there is no compaction or
  free-space bitmap.
- Free space = `capacity - total_sectors` (from FS_GET_INFO).
- This mirrors early FAT/CP/M behavior: simple, predictable, and sufficient
  for an educational OS with a small number of files.

## 9. Shell Integration — The `dir` Command

The shell's `dir` command demonstrates the complete filesystem stack:

```
User types "dir"
  → Shell calls INT 0x81, AH=0x01 (FS_LIST_FILES)
  → FS.SYS copies cached directory to shell's buffer
  → Shell parses entries and displays:

  Volume: MNFS v1     4 file(s)

  Name          Type   Sectors     Bytes
  ─────────────────────────────────────────
  LOADER   BIN  SYS        2       1024
  FS       BIN  SYS        2       1024
  KERNEL   BIN  SYS        4       2048
  SHELL    BIN  EXE       10       5120
  ─────────────────────────────────────────
  4 file(s)                        9216 bytes
```

### 9.1 Implementation Flow

```nasm
cmd_dir:
    ; 1. Call FS_LIST_FILES to get directory data
    mov ah, FS_LIST_FILES       ; 0x01
    mov bx, dir_buffer          ; 512-byte buffer in shell's data area
    push ds
    pop es                      ; ES = DS (shell segment)
    int 0x81

    ; 2. Parse header: file_count at offset 5
    mov cl, [dir_buffer + 5]    ; CL = file count

    ; 3. Loop through entries starting at offset 32
    mov si, dir_buffer + 32     ; SI → first entry
    ; ... format and print each entry ...
```

## 10. Build Pipeline Changes

### 10.1 New Binary: FS.SYS

`build.ps1` assembles `src/fs/fs.asm` into `build/boot/FS.SYS`.  The build
uses the same `-I src/include/` flag as all other binaries.

### 10.2 create-disk.ps1 Changes

The disk creation script is significantly simplified:

**Before (v0.5.0):** Hardcoded sector offsets for each binary.

```powershell
$LoaderPartOff = 4
$KernelPartOff = 20
$ShellPartOff  = 36
```

**After (v0.6.0):** Files are packed contiguously after the directory.
The script generates the MNFS directory sector from the actual file sizes.

```powershell
# Directory always at partition sector 2
$dirSector = 2

# Pack files contiguously starting at sector 3
$nextSector = 3
foreach ($file in $files) {
    $entry.start_sector = $nextSector
    $entry.size_sectors = $file.Length / 512
    $nextSector += $entry.size_sectors
}

# Generate and write the 512-byte MNFS directory sector
```

### 10.3 Include File Changes

| File              | Change                                            |
|-------------------|---------------------------------------------------|
| `disk.inc`        | **Removed** — replaced by `mnfs.inc`               |
| `mnfs.inc` (new)  | `MNFS_DIR_SECTOR equ 2`, entry offsets, attr flags |
| `find_file.inc` (new) | Bootstrap directory lookup subroutine          |
| `load_binary.inc` | Unchanged — still takes partition offset from caller |
| `syscalls.inc`    | Add `FS_*` constants for INT 0x81 functions        |

## 11. Boot Chain with MNFS

### 11.1 Complete Boot Sequence

```
MBR (sector 0)
  │  Scans partition table, finds active partition
  │  Loads VBR from partition start
  │
  ▼
VBR (partition sector 0-1, at 0x7C00)
  │  Populates BIB (drive, partition LBA)
  │  Reads MNFS directory from partition sector 2
  │  find_file("LOADER  SYS") → gets start sector
  │  load_mnex(start_sector) → loads LOADER at 0x0800
  │
  ▼
LOADER (at 0x0800)
  │  Enables A20 gate (3 fallback methods)
  │  Reads MNFS directory from partition sector 2
  │  find_file("KERNEL  SYS") → gets start sector
  │  load_mnex(start_sector) → loads KERNEL at 0x5000
  │
  ▼
KERNEL (at 0x5000)
  │  Installs INT 0x80 syscall handler
  │  Reads MNFS directory from partition sector 2
  │  find_file("FS      SYS") → loads FS.SYS at 0x0800
  │  Calls FS.SYS init → installs INT 0x81
  │  find_file("SHELL   SYS") → loads SHELL at 0x3000
  │
  ▼
SHELL (at 0x3000)
     User-mode executable
     Uses INT 0x80 for kernel services
     Uses INT 0x81 for filesystem services
     "dir" command lists files via INT 0x81
```

### 11.2 What's Still Hardcoded?

| Constant          | Value | Why it must be hardcoded                    |
|-------------------|-------|---------------------------------------------|
| VBR location      | 0     | Inherent — partition starts at sector 0     |
| Directory sector  | 2     | Bootstrap anchor — VBR must find it somehow |

Everything else — LOADER, FS.SYS, KERNEL, SHELL positions — is now dynamic,
determined by the directory table at build time.

## 12. Limitations

| Limitation                  | Rationale                                    |
|-----------------------------|----------------------------------------------|
| 15 files maximum            | 1-sector directory, 32-byte entries          |
| No subdirectories           | Flat filesystem by design                    |
| No fragmentation            | Files must be contiguous sectors             |
| No defragmentation          | Deleted file sectors leave unreclaimable gaps |
| No timestamps               | Unnecessary for educational OS               |
| 8.3 filenames only          | FAT-compatible, simple to implement          |
| Single directory read       | No multi-sector directory support            |

---

## 13. Future Extensions

These are not planned for v0.6.0 but the design accommodates them:

| Extension                | How it could be added                          |
|--------------------------|------------------------------------------------|
| Defragmentation          | Compact directory and data area to reclaim gaps |
| Long filenames           | Use reserved bytes in entry, or chain entries  |
| Multiple directories     | Multi-sector directory, parent pointers        |
| File allocation table    | Replace contiguous layout with FAT-style chain |
| Disk-based file creation | Shell `mkfile` command, dynamic sector allocation |
| Volume label             | Store in directory header reserved bytes       |
| File timestamps          | Use reserved bytes in directory entries        |
