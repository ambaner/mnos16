# MNEX Binary Format & Toolchain Specification

The MNOS Executable (MNEX) format is a custom binary format for mini-os.  This
document specifies the header layout for all binary types, the toolchain used
to produce them, the build pipeline from source to bootable disk, and the
evolution path from 16-bit to 64-bit.

---

## 1. Design Goals

1. **Self-describing** — Every binary carries a header that tells the loader
   what it is, how big it is, where it runs, and how to execute it.  No
   hardcoded sizes or addresses in the loader.

2. **Simple to parse** — The header must be parseable in ~10 assembly
   instructions.  No variable-length fields, no pointer chasing, no section
   tables.

3. **Mode-aware** — The same header format works for 16-bit, 32-bit, and
   64-bit binaries.  The loader reads the CPU mode field and validates it
   against the current CPU state before executing the binary.

4. **Unified** — One header format for all binaries.  No "legacy" vs "full"
   split.  A 16-bit executable and a 64-bit kernel binary use the exact same
   32-byte header structure.  The loader has one code path, not two.

5. **Toolchain-independent** — The header is prepended by the build system,
   not generated by the assembler or compiler.  This means any tool that
   produces flat binary output can be wrapped in an MNEX header.

---

## 2. Binary Types

mini-os defines the following binary types, each identified by a unique 4-byte
magic number.  **All types use the same 32-byte MNEX header** (§3):

| Magic | Type | Description |
|-------|------|-------------|
| `MNOS` | VBR | Volume Boot Record (partition boot sector) |
| `MNLD` | Loader | Stage-2 loader (A20, mode switch, binary loading) |
| `MNKN` | Kernel | OS kernel (interrupt handling, syscalls, drivers) |
| `MNEX` | Executable | User-mode executable (loaded and run by the kernel) |

Every magic is valid at **every `cpu_mode`** (0, 1, or 2).  The magic
identifies the binary's **role** — what it does in the system — while
`cpu_mode` (§3, offset 6) identifies the **bit depth** it was compiled for.
The two fields are orthogonal:

```
Magic  = WHAT the binary is    (VBR, Loader, Kernel, Executable)
cpu_mode = HOW it was built     (16-bit, 32-bit, 64-bit)
```

For example:

| Binary | Magic | cpu_mode | Meaning |
|--------|-------|----------|---------|
| 16-bit kernel | `MNKN` | 0 | Real-mode kernel (uses BIOS interrupts) |
| 16-bit shell | `MNEX` | 0 | Real-mode shell (loaded by 16-bit kernel) |
| 32-bit kernel | `MNKN` | 1 | Protected-mode kernel (own drivers, IDT) |
| 32-bit app | `MNEX` | 1 | Protected-mode executable |
| 64-bit kernel | `MNKN` | 2 | Long-mode kernel (paging, 64-bit addressing) |
| 64-bit app | `MNEX` | 2 | Long-mode executable |

The shell is **not a separate binary type** — it is a user-mode executable
(`MNEX`) loaded by the kernel, just like `cmd.exe` or `bash` in modern
operating systems.  This keeps the architecture clean: the kernel is the only
component that manages hardware and loads executables.

The loader or kernel **rejects any binary whose `cpu_mode` does not match**
the current CPU mode.  Mode transitions are one-way upgrades only (see §4).

## 3. MNEX Header (Universal, 32 bytes)

**Every** mini-os binary (except the MBR, which has no header) uses this
single unified header format.  This applies equally to 16-bit boot binaries,
32-bit kernel binaries, and future 64-bit binaries.

```
┌────────┬───────┬──────────────┬──────────────────────────────────────┐
│ Offset │ Size  │ Field        │ Description                          │
├────────┼───────┼──────────────┼──────────────────────────────────────┤
│ 0      │ 4 B   │ magic        │ Binary type magic (see §2)           │
│ 4      │ 2 B   │ version      │ Header format version (currently 1)  │
│ 6      │ 1 B   │ cpu_mode     │ Target CPU mode:                     │
│        │       │              │   0 = 16-bit real mode               │
│        │       │              │   1 = 32-bit protected mode          │
│        │       │              │   2 = 64-bit long mode               │
│ 7      │ 1 B   │ flags        │ Bit flags:                           │
│        │       │              │   bit 0: has BSS section             │
│        │       │              │   bit 1: has relocation data         │
│        │       │              │   bit 2: requires FPU/SSE            │
│        │       │              │   bits 3–7: reserved (must be 0)     │
│ 8      │ 4 B   │ load_addr    │ Target load address (linear)         │
│ 12     │ 4 B   │ entry_offset │ Entry point (offset from load_addr)  │
│ 16     │ 4 B   │ code_size    │ Code + initialized data size (bytes) │
│ 20     │ 4 B   │ bss_size     │ Uninitialized data (zero-fill after  │
│        │       │              │ code_size bytes; 0 if flag bit 0 = 0)│
│ 24     │ 4 B   │ stack_size   │ Recommended stack size (bytes)       │
│        │       │              │ Loader allocates this above BSS      │
│ 28     │ 4 B   │ checksum     │ CRC-32 of code region (0 = skip)    │
│ 32     │ ...   │ (code/data)  │ Binary content starts here           │
└────────┴───────┴──────────────┴──────────────────────────────────────┘
Total header: 32 bytes
```

### 3.1 VBR Header — Special Case

The VBR is unique because the BIOS/MBR jumps to byte 0 of the partition.
The CPU must execute a jump instruction immediately — it cannot encounter
data bytes first.  The VBR therefore wraps the MNEX header in a boot sector
preamble:

```
┌────────┬───────┬──────────────┬──────────────────────────────────────┐
│ Offset │ Size  │ Field        │ Description                          │
├────────┼───────┼──────────────┼──────────────────────────────────────┤
│ 0      │ 2 B   │ jmp short    │ EB xx — jump past header             │
│ 2      │ 1 B   │ nop          │ 90 — standard boot sector padding    │
│ 3      │ 4 B   │ magic        │ 'MNOS'                               │
│ 7      │ 2 B   │ version      │ 1                                    │
│ 9      │ 1 B   │ cpu_mode     │ 0 (16-bit)                           │
│ 10     │ 1 B   │ flags        │ 0                                    │
│ 11     │ 4 B   │ load_addr    │ 0x00007C00                           │
│ 15     │ 4 B   │ entry_offset │ offset to vbr_code                   │
│ 19     │ 4 B   │ code_size    │ VBR total size in bytes              │
│ 23     │ 4 B   │ bss_size     │ 0                                    │
│ 27     │ 4 B   │ stack_size   │ 0 (stack inherited from MBR)         │
│ 31     │ 4 B   │ checksum     │ 0 (disabled)                         │
│ 35     │ 4 B   │ part_lba     │ Partition start LBA (build-stamped)  │
│ 39     │ ...   │ (VBR code)   │                                      │
│ 510    │ 2 B   │ boot_sig     │ 0xAA55                               │
└────────┴───────┴──────────────┴──────────────────────────────────────┘
```

The MBR reads the MNEX header starting at offset 3 (after the jmp+nop).
The `part_lba` field at offset 35 is VBR-specific extra data placed after
the standard 32-byte header body (which starts at offset 3, ending at
offset 34).

### 3.2 Non-VBR Binaries (LOADER, SHELL, KERNEL, etc.)

All other binaries start directly with the MNEX header at byte 0.  The
loading stage reads the header, validates the magic and cpu_mode, then jumps
to `load_addr + entry_offset`:

```
Byte 0:  magic[0]    ← header starts immediately
Byte 4:  version
Byte 6:  cpu_mode
...
Byte 32: (first instruction of code, if entry_offset = 32)
```

### 3.3 Field Details

**magic** (4 bytes) — Identifies the binary type.  The loader checks this
before processing any other fields.  If the magic is unrecognized, the binary
is rejected with an error message.

**version** (2 bytes) — Header format version.  Currently 1.  If a future
version adds fields after offset 28, the version number increases and the
loader can handle both old and new formats.

**cpu_mode** (1 byte) — The CPU mode the binary was built for.  This is the
**primary validation gate** — the loader compares this against the current
CPU state and rejects mismatches (see §4).  Values:

| Value | Mode | Required state before jump |
|-------|------|---------------------------|
| 0 | 16-bit real mode | Default at boot; no setup needed |
| 1 | 32-bit protected mode | GDT loaded, CR0.PE set, A20 enabled |
| 2 | 64-bit long mode | All of 32-bit + page tables + EFER.LME + CR0.PG |

**flags** (1 byte) — Optional capabilities:

- **Bit 0 (has_bss)**: If set, the loader zero-fills `bss_size` bytes
  starting at `load_addr + code_size`.  This is how C globals initialized to
  zero work (`static int count;` goes in BSS).  For 16-bit NASM binaries,
  this is typically 0 (NASM handles its own zeroing via `times`).

- **Bit 1 (has_reloc)**: Reserved for future use.  If set, relocation data
  follows the code section.  Not implemented in v1.

- **Bit 2 (needs_fpu)**: If set, the loader enables the FPU/SSE before
  jumping to the entry point (set CR0.EM=0, CR4.OSFXSR=1).

**load_addr** (4 bytes) — The linear (physical) address where the binary
should be placed in memory.  Examples:

| Binary | load_addr | Rationale |
|--------|-----------|-----------|
| LOADER.SYS | 0x00000800 | Below 1 MB, accessible in real mode |
| SHELL.SYS | 0x00003000 | Below 1 MB, accessible in real mode |
| KERNEL.SYS | 0x00100000 | 1 MB mark, start of extended memory |
| KERNEL64.BIN | 0x00100000 | Same (identity-mapped) |

**entry_offset** (4 bytes) — Byte offset from `load_addr` to the first
instruction to execute.  For NASM binaries this is typically 32 (immediately
after the 32-byte header).  For C binaries it may differ if the entry point
function is not at the start of the `.text` section.  The loader jumps to
`load_addr + entry_offset`.

**code_size** (4 bytes) — Total bytes of the binary image on disk (header +
code + initialized data).  The loader reads exactly this many bytes from disk.
For 16-bit binaries, this must be a multiple of 512 (sector-aligned).

**bss_size** (4 bytes) — Bytes of uninitialized data to zero-fill in memory
immediately after the loaded image.  The C compiler puts zero-initialized
globals here.  If `flags.has_bss` is 0, this field is ignored and should be 0.
For 16-bit NASM binaries, this is 0 (the `times` padding already zeroes the
data section).

**stack_size** (4 bytes) — Suggested stack allocation.  The loader places the
stack at `load_addr + code_size + bss_size + stack_size` (top of stack, grows
down).  If 0, the loader uses the inherited stack (appropriate for 16-bit
binaries that share the boot-time stack at 0x7C00).

**checksum** (4 bytes) — CRC-32 of the code region (bytes 32 through
code_size - 1).  If 0, the loader skips the integrity check.  Useful for
catching disk corruption on larger binaries.

### 3.4 Memory Layout After Loading

```
load_addr
    │
    ├── [0 .. 31]                    MNEX header (32 bytes)
    ├── [32 .. code_size-1]          Code + initialized data
    ├── [code_size .. code_size+bss_size-1]   BSS (zeroed by loader)
    │
    ├── (stack grows downward)
    │
    └── [code_size + bss_size + stack_size]   Stack top (ESP/RSP initial)
```

For 16-bit binaries where `bss_size = 0` and `stack_size = 0`, the memory
layout is simply the header followed by code — identical to the current
behavior, just with a richer header.

### 3.5 Example Headers

**16-bit Executable / Shell (MNEX, cpu_mode=0)**:
```
4D 4E 45 58   magic:        'MNEX'
01 00         version:      1
00            cpu_mode:     16-bit real mode
00            flags:        0 (no BSS, no reloc)
00 30 00 00   load_addr:    0x00003000
20 00 00 00   entry_offset: 0x00000020 (32 = header size)
00 14 00 00   code_size:    0x00001400 (5120 bytes = 10 sectors)
00 00 00 00   bss_size:     0
00 00 00 00   stack_size:   0 (inherits boot stack)
00 00 00 00   checksum:     0 (disabled)
```

**32-bit Kernel (MNKN, cpu_mode=1)**:
```
4D 4E 4B 4E   magic:        'MNKN'
01 00         version:      1
01            cpu_mode:     32-bit protected mode
01            flags:        has_bss = 1
00 00 10 00   load_addr:    0x00100000 (1 MB)
20 00 00 00   entry_offset: 0x00000020 (32 = header size)
00 40 00 00   code_size:    0x00004000 (16 KB)
00 10 00 00   bss_size:     0x00001000 (4 KB)
00 40 00 00   stack_size:   0x00004000 (16 KB)
00 00 00 00   checksum:     0 (disabled)
```

**64-bit Kernel (MNKN, cpu_mode=2)**:
```
4D 4E 4B 4E   magic:        'MNKN'
01 00         version:      1
02            cpu_mode:     64-bit long mode
03            flags:        has_bss + needs_fpu
00 00 10 00   load_addr:    0x00100000 (1 MB, identity-mapped)
20 00 00 00   entry_offset: 0x00000020
00 80 00 00   code_size:    0x00008000 (32 KB)
00 20 00 00   bss_size:     0x00002000 (8 KB)
00 00 01 00   stack_size:   0x00010000 (64 KB)
A3 B5 C7 D9   checksum:     CRC-32 of code region
```

### 3.6 Current Runtime Format: v2 Compact Header (16-bit, v0.9.14)

The 32-byte universal header (§3) is the **target format** for future versions
(32-bit/64-bit).  For the current 16-bit implementation, all system modules and
user programs use a **compact v2 header** optimized for minimal loader complexity:

```
┌────────┬───────┬──────────────┬──────────────────────────────────────┐
│ Offset │ Size  │ Field        │ Description                          │
├────────┼───────┼──────────────┼──────────────────────────────────────┤
│ 0      │ 4 B   │ magic        │ Module/program type ('MNFS', 'MNMM', │
│        │       │              │ 'MNSH', 'MNEX')                      │
│ 4      │ 2 B   │ sector_count │ Total file size in 512-byte sectors  │
│ 6      │ 2 B   │ flags        │ Bit 0: MNEX_V2_FLAG_RELOC (=0x0001) │
│        │       │              │ If set: has relocation table         │
│ 8      │ 2 B   │ reloc_count  │ Number of 16-bit relocation entries  │
│ 10     │ 2 B   │ entry_offset │ Code entry point (bytes from load    │
│        │       │              │ base to first instruction)           │
│ 12     │ N×2 B │ reloc_table  │ Array of file-relative offsets; each │
│        │       │              │ points to a 16-bit word to patch     │
│ 12+N×2 │ ...   │ code/data    │ Binary content (assembled at ORG 0)  │
└────────┴───────┴──────────────┴──────────────────────────────────────┘
Total header: 12 + (reloc_count × 2) bytes (variable)
```

**Key differences from the 32-byte universal header:**

| Aspect | Universal (§3) | v2 Compact (current) |
|--------|---------------|---------------------|
| Header size | Fixed 32 bytes | Variable (12 + relocs) |
| Fields | 10 fields (32/64-bit ready) | 5 fields (16-bit focused) |
| Relocation | Flag bit 1 reserved | Fully implemented |
| Load address | Explicit `load_addr` field | Determined by loader at runtime |
| BSS/Stack | Explicit fields | Not needed (NASM handles via `times`) |
| CPU mode | Explicit field | Implicit (all 16-bit) |

**Migration path**: When mini-os transitions to 32-bit mode, the universal
32-byte header will replace the v2 compact header.  The `flags.has_reloc`
bit (bit 1) in the universal header will indicate that a relocation table
follows the 32-byte header, extending the same concept.

**Runtime relocation algorithm** (applied by kernel for modules, shell for programs):
```
for i = 0 to reloc_count - 1:
    offset = reloc_table[i]           ; file-relative offset
    word_ptr = load_base + offset     ; physical address of word to patch
    [word_ptr] += load_base           ; add actual load address
entry_point = load_base + entry_offset
```

**Pre-biasing** (applied at build time by `pack_module.py`):
- Code is assembled at ORG 0, so all label references are zero-based offsets
  into the raw code section
- `pack_module.py` prepends the header + reloc_table (prefix_size bytes)
- Each word targeted by a relocation entry gets `prefix_size` added
- At load time, the loader adds `load_base` to complete the address

## 4. CPU Mode Validation

The loader maintains awareness of the current CPU mode and validates every
binary before execution.  This prevents silent crashes from mode mismatches
(e.g., trying to execute 32-bit instructions while still in 16-bit mode).

### 4.1 Validation Logic

```
Load binary from disk
    │
    ├── Read magic (offset 0) → verify known type
    │
    ├── Read cpu_mode (offset 6) → compare to current CPU mode
    │       │
    │       ├── MATCH → proceed to load
    │       │
    │       ├── UPGRADE POSSIBLE → loader switches mode, then loads
    │       │   (e.g., current=16-bit, binary=32-bit, loader has GDT ready)
    │       │
    │       └── MISMATCH → error and halt
    │               "Error: binary requires 32-bit mode,
    │                current mode is 16-bit"
    │
    ├── Read load_addr, code_size → load binary to correct address
    │
    ├── If flags.has_bss → zero-fill BSS region
    │
    ├── If stack_size > 0 → set SP/ESP/RSP
    │
    └── Jump to load_addr + entry_offset
```

### 4.2 Mode Transition Rules

The loader follows strict rules about which transitions are permitted:

| Current Mode | Binary cpu_mode | Action |
|-------------|-----------------|--------|
| 16-bit (0) | 16-bit (0) | ✅ Load and jump directly |
| 16-bit (0) | 32-bit (1) | ✅ Set up GDT, switch to PM, then jump |
| 16-bit (0) | 64-bit (2) | ✅ Set up GDT + page tables, switch to LM, then jump |
| 32-bit (1) | 32-bit (1) | ✅ Load and jump directly |
| 32-bit (1) | 16-bit (0) | ❌ **Error**: cannot downgrade to real mode |
| 32-bit (1) | 64-bit (2) | ✅ Set up page tables, switch to LM, then jump |
| 64-bit (2) | 64-bit (2) | ✅ Load and jump directly |
| 64-bit (2) | 32-bit (1) | ❌ **Error**: cannot downgrade from long mode |
| 64-bit (2) | 16-bit (0) | ❌ **Error**: cannot downgrade from long mode |

**Key principle**: Mode transitions are **one-way upgrades**.  Once the CPU
enters protected mode, it does not return to real mode.  Once in long mode,
it does not return to protected mode.  This matches how real operating systems
work and avoids the complexity and fragility of mode downgrading.

### 4.3 Error Messages

When a mode mismatch is detected, the loader prints a clear diagnostic:

```
MNEX Error: mode mismatch
  Binary:  MNKN (KERNEL.SYS)
  Requires: 32-bit protected mode
  Current:  16-bit real mode
  Action:   Cannot load — loader does not support mode switch

  (or, if mode switch is supported but the binary wants a downgrade):

  Action:   Cannot downgrade from 32-bit to 16-bit
```

### 4.4 Implementation in LOADER.SYS (16-bit)

```nasm
; After loading the first sector of a binary and verifying magic:
    mov al, [load_addr + 6]         ; Read cpu_mode field
    cmp al, 0                       ; 16-bit?
    je .mode_ok_16                  ;   → load and jump directly

    cmp al, 1                       ; 32-bit?
    je .mode_switch_32              ;   → set up GDT, switch, then jump

    cmp al, 2                       ; 64-bit?
    je .mode_switch_64              ;   → GDT + paging, switch, then jump

    ; Unknown mode
    mov si, msg_bad_mode
    call puts
    jmp .halt

.mode_ok_16:
    ; Load remaining sectors, jump to entry point (16-bit)
    ; ... (existing load logic) ...
    jmp far [load_addr + 12]        ; entry_offset

.mode_switch_32:
    ; Load all sectors to bounce buffer
    ; Set up GDT
    ; Copy binary to load_addr (may be above 1 MB)
    ; Zero BSS if flagged
    ; Set ESP from stack_size
    ; Switch to protected mode (cli, lgdt, cr0.pe, far jmp)
    ; Jump to load_addr + entry_offset
    ; ... (new code for v0.5.0) ...

.mode_switch_64:
    ; Everything from .mode_switch_32, plus:
    ; Check CPUID for long mode support
    ; Build page tables
    ; Enable PAE, EFER.LME, paging
    ; Far jump to 64-bit segment
    ; ... (future) ...
```

### 4.5 Implementation in 32-bit Kernel (Loading Executables)

Once the kernel is running in 32-bit mode, it may load additional MNEX
binaries (e.g., user programs).  The kernel's loader performs the same
validation:

```c
/* mnex.h — MNEX header structure */
#include <stdint.h>

#define MNEX_MODE_16  0
#define MNEX_MODE_32  1
#define MNEX_MODE_64  2

typedef struct __attribute__((packed)) {
    char     magic[4];
    uint16_t version;
    uint8_t  cpu_mode;
    uint8_t  flags;
    uint32_t load_addr;
    uint32_t entry_offset;
    uint32_t code_size;
    uint32_t bss_size;
    uint32_t stack_size;
    uint32_t checksum;
} mnex_header_t;

/* Validate an MNEX header against current CPU mode */
int mnex_validate(const mnex_header_t *hdr, uint8_t current_mode) {
    if (hdr->version != 1) return -1;          /* Unknown version */
    if (hdr->cpu_mode > MNEX_MODE_64) return -2; /* Invalid mode */
    if (hdr->cpu_mode < current_mode) return -3;  /* Downgrade rejected */
    return 0;  /* OK */
}
```

## 5. Toolchain

### 5.1 Why This Combination

| Decision | Chosen | Why |
|----------|--------|-----|
| **16-bit assembler** | NASM | Industry standard for x86 boot code; `-f bin` produces flat binaries; auto-downloaded by build system |
| **32/64-bit compiler** | Clang (LLVM) | Cross-compiles to `i686-elf` / `x86_64-elf` freestanding targets; pre-installed on GitHub Actions runners; no cross-compiler build step; produces standard ELF objects |
| **Linker** | ld.lld (LLVM) | Ships with Clang; understands linker scripts; produces ELF executables from mixed ASM+C objects; cross-platform (Windows/Linux/macOS) |
| **Binary extraction** | llvm-objcopy | Converts ELF → flat binary; ships with LLVM; same as `objcopy -O binary` |
| **Header stamping** | PowerShell script | Prepends MNEX header to flat binary; consistent with existing build tooling (create-disk.ps1, build.ps1) |
| **Build orchestration** | PowerShell 7 + batch wrapper | Already proven; cross-platform (Windows + CI); no Make/CMake dependency |

### 5.2 Alternatives Considered

| Alternative | Verdict | Reason |
|-------------|---------|--------|
| **NASM for everything** | ❌ Rejected for 32/64-bit | Assembly-only kernel is unreadable; a `puts` with scrolling is 100+ lines of ASM vs 20 lines of C; educational value diminishes past the boot stages |
| **GCC cross-compiler** | ❌ Rejected | Not available on GitHub Actions; requires 30-minute build from source or Docker; painful local setup on Windows |
| **Zig** | Considered | Excellent cross-compilation, small download (~45 MB), includes C compiler (`zig cc`); less conventional for OS tutorials; could revisit if Clang setup proves problematic |
| **Rust** | ❌ Rejected | Custom target JSON, `no_std` boilerplate, steep learning curve for kernel work; over-engineered for an educational project |
| **ELF format** | ❌ Rejected for on-disk | ELF parsing in a bootloader requires ~200 lines of code (section headers, program headers, string tables); MNEX achieves the same with 10 lines |
| **PE format** | ❌ Rejected | Windows-specific, even more complex than ELF; no benefit for bare-metal |
| **No custom format** | ❌ Rejected | Hardcoded sizes and addresses are fragile; self-describing headers have proven their value since v0.2.1 |


## 6. Build Pipeline

### 6.1 Phase 1 — 16-bit Binaries (NASM + wrap-mnex.ps1)

```
src/boot/mbr.asm ─── nasm -f bin ──→ mbr.bin (512 B, no MNEX header — BIOS std)

src/boot/vbr.asm ─── nasm -f bin ──→ vbr.raw ──→ wrap-mnex.ps1 ──→ vbr.bin
src/loader/loader.asm ─ nasm -f bin → loader.raw → wrap-mnex.ps1 → LOADER.SYS
src/kernel/kernel.asm ─ nasm -f bin → kernel.raw → wrap-mnex.ps1 → KERNEL.SYS
src/shell/shell.asm ─── nasm -f bin → shell.raw  → wrap-mnex.ps1 → SHELL.SYS
                                                        │
                                                  All binaries have
                                                  unified 32-byte MNEX
                                                  header (cpu_mode = 0)
```

Note: The VBR binary includes the jmp+nop preamble in its NASM source; its
MNEX header is embedded starting at offset 3 (see §3.1).  The wrap step for
VBR stamps the header fields into the pre-allocated space in the assembly
output rather than prepending.

For LOADER and SHELL, `wrap-mnex.ps1` prepends the 32-byte header to the
flat binary output from NASM.  The NASM source uses `[ORG load_addr + 32]`
to account for the header.

### 6.2 Phase 2 — 32-bit Kernel (NASM + Clang)

```
src/kernel/entry.asm ──── nasm -f elf32 ──→ entry.o ──┐
src/kernel/main.c ─────── clang --target ──→ main.o ──┤
src/kernel/vga.c ──────── clang --target ──→ vga.o ───┤
src/kernel/keyboard.c ─── clang --target ──→ kb.o ────┤
src/kernel/idt.c ──────── clang --target ──→ idt.o ───┘
                                                       │
                                                  ld.lld -T kernel.ld
                                                       │
                                                  kernel.elf
                                                       │
                                                  llvm-objcopy -O binary
                                                       │
                                                  kernel.raw (flat binary)
                                                       │
                                                  tools/wrap-mnex.ps1
                                                       │
                                                  build/boot/KERNEL.SYS
                                                  (MNEX header + flat binary)
```

**Clang flags**:
```
clang                          \
    --target=i686-elf          \  Target: 32-bit x86, no OS
    -ffreestanding             \  No standard library assumptions
    -fno-builtin               \  Don't replace code with libc calls
    -nostdlib                  \  Don't link standard C library
    -nostdinc                  \  Don't search system include paths
    -fno-stack-protector       \  No stack canaries (no OS to handle them)
    -mno-red-zone              \  Don't use red zone (unsafe in kernel)
    -fno-pic                   \  No position-independent code
    -O2                        \  Optimise (smaller, faster code)
    -Wall -Wextra              \  All warnings
    -c kernel/main.c           \  Compile only (don't link)
    -o main.o                     Output object file
```

**Linker script** (`src/kernel/kernel.ld`):
```ld
/* kernel.ld — Controls kernel memory layout */
OUTPUT_FORMAT(elf32-i386)
ENTRY(_start)

SECTIONS
{
    /* Skip the 32-byte MNEX header (added by wrap-mnex.ps1 post-link) */
    . = 0x100020;              /* load_addr (0x100000) + header (0x20) */

    .text : {
        *(.text)               /* All code sections */
    }

    .rodata : {
        *(.rodata*)            /* Read-only data (string literals, etc.) */
    }

    .data : {
        *(.data)               /* Initialized global variables */
    }

    _bss_start = .;
    .bss : {
        *(.bss)                /* Uninitialized globals (zeroed) */
        *(COMMON)              /* C "common" symbols */
    }
    _bss_end = .;

    _kernel_end = .;
}
```

**NASM entry point** (`src/kernel/entry.asm`):
```nasm
; entry.asm — 32-bit kernel entry point
; Called by LOADER after mode switch.  Sets up C runtime and calls main.

[BITS 32]
[EXTERN kernel_main]            ; C function in main.c
[EXTERN _bss_start]             ; Linker symbols
[EXTERN _bss_end]
[GLOBAL _start]

_start:
    ; Zero out BSS section (C expects uninitialized globals = 0)
    mov edi, _bss_start
    mov ecx, _bss_end
    sub ecx, edi
    xor al, al
    rep stosb

    ; Call C kernel
    call kernel_main

    ; If kernel_main returns, halt
.halt:
    cli
    hlt
    jmp .halt
```

### 6.3 Phase 3 — 64-bit Kernel (Future)

Same pipeline as Phase 2, but:
- `--target=x86_64-elf` instead of `i686-elf`
- `nasm -f elf64` instead of `elf32`
- Linker script uses `OUTPUT_FORMAT(elf64-x86-64)`
- Magic changes from `MNKN` (same) — only `cpu_mode` changes from 1 to 2
- `cpu_mode` field = 2

### 6.4 User-Mode C Executables (Phase 2+)

Starting in Phase 2, user-mode executables (MNEX binaries) can be written
in C.  The build pipeline mirrors the kernel pipeline, but produces MNEX
(not MNKN) binaries and uses a user-mode linker script with the appropriate
load address:

```
src/apps/hello.c ──── clang --target=i686-elf ──→ hello.o
                                                     │
                                                ld.lld -T user.ld
                                                     │
                                                hello.elf
                                                     │
                                                llvm-objcopy -O binary
                                                     │
                                                hello.raw (flat binary)
                                                     │
                                                tools/wrap-mnex.ps1 -Magic MNEX -CpuMode 1
                                                     │
                                                build/apps/hello.bin
                                                (MNEX header + flat binary)
```

C executables call kernel syscalls via inline assembly wrappers in
`mnos_syscall.h`.  The application never executes `int $0x80` directly —
it calls typed C functions like `print()`, `readkey()`, and `cls()`.

See [SYSTEM-CALLS.md §7.4](SYSTEM-CALLS.md#74-c-language-bindings) for the
complete `mnos_syscall.h` header, example programs, and the 16→32-bit
transition strategy.

> **16-bit limitation**: Clang cannot generate 16-bit real-mode code.  The
> 16-bit kernel and shell must remain in NASM assembly.  C support begins
> with the 32-bit protected-mode kernel (Phase 2).

### 6.5 wrap-mnex.ps1 — Header Stamper

This script takes a flat binary and prepends the 32-byte MNEX header:

```powershell
param(
    [string]$InputFile,          # kernel.raw (flat binary from objcopy)
    [string]$OutputFile,         # KERNEL.SYS (MNEX-wrapped)
    [string]$Magic = "MNKN",    # 4-byte magic
    [int]$CpuMode = 1,          # 0=16, 1=32, 2=64
    [uint32]$LoadAddr = 0x100000,
    [uint32]$EntryOffset = 0x20, # 32 bytes (header size)
    [uint32]$BssSize = 0,
    [uint32]$StackSize = 0x4000  # 16 KB default
)

$code = [System.IO.File]::ReadAllBytes($InputFile)
$header = [byte[]]::new(32)

# Magic (4 bytes)
[System.Text.Encoding]::ASCII.GetBytes($Magic, 0, 4, $header, 0)
# Version (2 bytes, little-endian)
$header[4] = 1; $header[5] = 0
# CPU mode (1 byte)
$header[6] = $CpuMode
# Flags (1 byte)
$header[7] = if ($BssSize -gt 0) { 1 } else { 0 }
# Load address (4 bytes LE)
[BitConverter]::GetBytes([uint32]$LoadAddr).CopyTo($header, 8)
# Entry offset (4 bytes LE)
[BitConverter]::GetBytes([uint32]$EntryOffset).CopyTo($header, 12)
# Code size (4 bytes LE) — includes header
[BitConverter]::GetBytes([uint32]($code.Length + 32)).CopyTo($header, 16)
# BSS size
[BitConverter]::GetBytes([uint32]$BssSize).CopyTo($header, 20)
# Stack size
[BitConverter]::GetBytes([uint32]$StackSize).CopyTo($header, 24)
# Checksum (4 bytes) — 0 = disabled
# header[28..31] already 0

# Write header + code
$output = $header + $code
[System.IO.File]::WriteAllBytes($OutputFile, $output)
```

## 7. Binary Catalog

### 7.1 Complete Binary Summary

```
Binary         Magic   Header     CPU   Load Addr   Produced By     Loaded By
─────────────  ──────  ─────────  ────  ──────────  ──────────────  ──────────
mbr.bin        (none)  N/A        16    0x7C00      NASM            BIOS
vbr.bin        MNOS    MNEX 32B   16    0x7C00      NASM+wrap       MBR
LOADER.SYS     MNLD    MNEX 32B   16    0x0800      NASM+wrap       VBR
KERNEL.SYS     MNKN    MNEX 32B   16*   0x5000      NASM+wrap       LOADER
FS.SYS         MNFS    MNEX 32B   16    0x0800      NASM+wrap       KERNEL
MM.SYS         MNMM    MNEX 32B   16    0x2800      NASM+wrap       KERNEL
SHELL.SYS      MNEX    MNEX 32B   16    0x3000      NASM+wrap       KERNEL†
app.mnx        MNEX    MNEX 32B   **    0x9000      NASM+wrap       SHELL
kernel32.bin   MNKN    MNEX 32B   32    0x100000    Clang+NASM+LD   LOADER
kernel64.bin   MNKN    MNEX 32B   64    0x100000    Clang+NASM+LD   LOADER
```

*\* The 16-bit kernel uses BIOS interrupts for hardware access.*
*\*\* Executables are built for the same cpu_mode as the kernel that loads them.*
*† SHELL.SYS uses `.SYS` because it is loaded by the kernel at boot into
system memory (0x3000) and runs for the lifetime of the OS.  User programs
loaded on demand use `.MNX`.*

All binaries (except MBR) share the identical 32-byte MNEX header.  The loader
uses one code path to parse any binary, then branches based on `cpu_mode`.

### 7.2 Disk Layout (All Phases)

```
Sector 0                → MBR (512 B, no header — BIOS standard)
Sectors 1–2047          → Reserved gap

Partition start (LBA 2048):
  Offset 0              → VBR (2 sectors, MNEX header with jmp preamble)
  Offset 4              → LOADER.SYS (up to 16 sectors, MNEX header)
  Offset 20             → KERNEL.SYS (up to 16 sectors, MNEX header, cpu_mode=0)
  Offset 36             → SHELL.SYS (up to 16 sectors, MNEX header, cpu_mode=0)
  Offset 52+            → Future executables / filesystem
```

### 7.3 Load Sequence

**Phase 1 (current v0.5.0)**:
```
BIOS → MBR (0x7C00)
         → VBR (0x7C00, via staging at 0x7E00)
              → LOADER.SYS (0x0800)
                   → KERNEL.SYS (0x5000, 16-bit, MNKN)
                        → SHELL.SYS (0x3000, 16-bit, MNEX)
                             → Interactive shell (mnos:\>)
```

**Phase 2 (32-bit kernel)**:
```
BIOS → MBR (0x7C00)
         → VBR (0x7C00)
              → LOADER.SYS (0x0800)
                   │
                   ├── Load KERNEL.SYS to bounce buffer (e.g., 0x10000)
                   │   ├── Verify MNKN magic + cpu_mode
                   │   ├── Read load_addr, code_size, bss_size from header
                   │   │
                   │   ├── Set up GDT
                   │   ├── Switch to protected mode (CR0.PE)
                   │   │
                   │   ├── Copy kernel from 0x10000 to 0x100000 (load_addr)
                   │   ├── Zero-fill BSS (code_size → code_size + bss_size)
                   │   ├── Set ESP = load_addr + code_size + bss_size + stack_size
                   │   │
                   │   └── JMP to load_addr + entry_offset
                   │
                   └── KERNEL.SYS running at 0x100000 in 32-bit mode
                        → Loads SHELL.SYS (MNEX, cpu_mode=1) as user-mode executable
                        → VGA driver, keyboard driver, shell in C
```

**Phase 3 (64-bit kernel)**:
```
Same as Phase 2, but LOADER additionally:
  ├── Checks CPUID for long mode support
  ├── Builds 4-level page tables (identity-map first N MB)
  ├── Enables PAE (CR4.PAE)
  ├── Enables long mode (EFER.LME via MSR)
  ├── Enables paging (CR0.PG)
  └── Far jumps to 64-bit code segment
```

### 8.3 C Coding Style for Kernel

The kernel C code follows these conventions for readability:

```c
/* ports.h — Inline assembly wrappers for x86 I/O port access */
#ifndef PORTS_H
#define PORTS_H

#include <stdint.h>    /* uint8_t, uint16_t — from compiler, not libc */

/* Read a byte from an I/O port */
static inline uint8_t inb(uint16_t port) {
    uint8_t val;
    __asm__ volatile ("inb %1, %0" : "=a"(val) : "Nd"(port));
    return val;
}

/* Write a byte to an I/O port */
static inline void outb(uint16_t port, uint8_t val) {
    __asm__ volatile ("outb %0, %1" : : "a"(val), "Nd"(port));
}

#endif
```

```c
/* vga.c — VGA text-mode driver */
#include "vga.h"
#include "ports.h"

#define VGA_BUFFER  ((volatile uint16_t*)0xB8000)
#define VGA_WIDTH   80
#define VGA_HEIGHT  25

static int cursor_x = 0;
static int cursor_y = 0;
static uint8_t color = 0x07;   /* Light grey on black */

void vga_putc(char c) {
    if (c == '\n') {
        cursor_x = 0;
        cursor_y++;
    } else {
        VGA_BUFFER[cursor_y * VGA_WIDTH + cursor_x] = (color << 8) | c;
        cursor_x++;
        if (cursor_x >= VGA_WIDTH) {
            cursor_x = 0;
            cursor_y++;
        }
    }
    if (cursor_y >= VGA_HEIGHT) {
        vga_scroll();
        cursor_y = VGA_HEIGHT - 1;
    }
    vga_update_cursor();
}
```

This is dramatically more readable than the equivalent assembly while producing
nearly identical machine code with `-O2`.

## 9. Migration from Legacy Headers

### 9.1 What Changes

The v0.4.0 binaries use a 6-byte legacy header (magic + sector count).  The
transition to unified MNEX headers requires:

| Binary | Current header | New header | Source change |
|--------|---------------|------------|---------------|
| VBR | 13 bytes (jmp+nop+MNOS+sectors+part_lba) | 35+ bytes (jmp+nop+MNEX 32B+part_lba) | Header area grows by ~22 bytes |
| LOADER | 6 bytes (MNLD + sectors) | 32 bytes (MNEX) | ORG adjusts by +26 bytes |
| KERNEL | (new binary) | 32 bytes (MNKN) | New 16-bit kernel, loads shell |
| SHELL | 6 bytes (MNSH + sectors) | 32 bytes (MNEX) | Magic changes from MNSH → MNEX |

### 9.2 What Stays the Same

- MBR has no header (BIOS standard) — unchanged
- Binary magics (MNOS, MNLD) — unchanged
- Disk layout (partition offsets 0, 4, 20) — unchanged
- Load addresses (0x7C00, 0x0800, 0x3000) — unchanged
- Build toolchain (NASM for 16-bit) — unchanged

### 9.3 Backward Compatibility

The MBR must be updated to parse the new VBR header layout (magic at offset 3
→ still at offset 3; sector count at offset 7 → now derived from `code_size`
at offset 19).  Similarly, VBR and LOADER must be updated to parse the full
32-byte header from the next binary.

This is a **breaking change** — v0.5.0 binaries are not interchangeable with
v0.4.0 binaries.  However, since all binaries are built together and deployed
as a single disk image, this is acceptable.

## 10. Security Considerations

The MNEX format includes minimal integrity checking:

| Threat | Mitigation | Status |
|--------|-----------|--------|
| Corrupted binary on disk | CRC-32 checksum field | Optional (v1) |
| Wrong binary type loaded | Magic number verification | Implemented |
| Binary too large for memory | code_size checked against available RAM | Planned |
| Buffer overflow in BSS | bss_size validated against memory map | Planned |
| Stack overflow | stack_size sets ESP correctly | Planned |
| Code execution in BSS/stack | Not mitigated (no paging/NX in v1) | Future (with paging) |

In a single-user educational OS with no networking, these are informational
rather than critical.  The checksum catches the most likely real-world issue:
disk corruption.

## 11. Versioning and Evolution

The MNEX format is versioned to allow future extension:

| Version | Header size | Changes |
|---------|-------------|---------|
| 1 (current spec) | 32 bytes | Initial format |
| 2 (future) | 48 bytes? | + dependency list, + min RAM requirement |
| 3 (future) | 64 bytes? | + digital signature, + compression flag |

Version 1 loaders ignore versions they don't understand and report an error.
Older binaries with version 1 headers always work with newer loaders.
